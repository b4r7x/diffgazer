title: How it works

You run one command, and a moment later a review opens in your browser. In between, **diffgazer starts an embedded local server and a UI on your own machine; nothing leaves it until you start a review.** No account, no upload, no dashboard you do not control. This page walks through what that command actually sets in motion.

## The pieces

There are four parts, and they all live on your computer.

The **CLI** is the `diffgazer` binary you install globally. When you run it, it decides which mode to start and brings everything else up.

The **embedded server** is a [Hono](https://hono.dev) backend that runs the review pipeline, talks to git, and reads your config. You do not install it separately. It is bundled straight into the `diffgazer` binary, so there is one thing to install and one process to run. When the server starts, it binds to `127.0.0.1`, which means it only listens for connections from your own machine.

The **web SPA** is the review interface. The same embedded server serves it, so the UI and the API share one origin on localhost. If you prefer to stay in the terminal, there is an Ink **terminal UI** instead. More on choosing between them below.

Your **provider** is the model you bring. diffgazer is provider-agnostic, so the AI work happens against the model and keys you supply. See [Providers and models](/app/concepts/providers-and-models) for how that is configured.

## The review pipeline

Once the UI is up and you start a review, the work runs as five steps, in order:

```text
diff  ->  context  ->  review  ->  enrich  ->  report
```

Here is what each step does:

- **diff** — gathering code changes.
- **context** — building the repo graph and summary.
- **review** — analyzing with lenses.
- **enrich** — adding git blame and context.
- **report** — synthesizing the final report.

The UI reports progress as each step starts and completes, so you can watch the review move through the pipeline. For a deeper look at what happens inside each step, see [The review pipeline](/app/concepts/review-pipeline).

## Web mode vs terminal UI

Web mode is the default. Run `diffgazer` with no flags and it starts the embedded server, serves the SPA, and opens your browser. It is the full-featured way to review, so reach for it unless you have a reason not to. The details live in [Web mode](/app/web).

The terminal UI is opt-in and still in beta. Pass `--tui` to start the Ink interface and review without leaving your shell. It is incomplete, so it is not the right choice for everyday work yet. If you want to try it anyway, see the [Terminal UI](/app/tui) section.

Both modes run the same pipeline against the same local server. The difference is only where you read the result.

## What runs locally

Everything you have read about so far runs on your machine: the CLI, the embedded server bound to `127.0.0.1`, the UI served over localhost, and your git history. The only thing that leaves your computer is the request you send to your chosen model provider. For exactly what stays local and what does not, see [Privacy](/app/concepts/privacy).
