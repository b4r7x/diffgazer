## Overview

A screen is a top-level React component rendered by the `ScreenRouter` based on the current route. Screens own their keyboard scope, footer shortcuts, and back-navigation behavior. Feature-specific logic is extracted into feature modules under `apps/cli/src/features/`.

## Screen Convention

Every screen should follow this pattern:

```tsx
import type { ReactElement } from "react";
import { useScope } from "../../hooks/use-scope.js";
import { usePageFooter } from "../../hooks/use-page-footer.js";
import { useBackHandler } from "../../hooks/use-back-handler.js";

export function ExampleScreen(): ReactElement {
  useScope("example");                              // 1. Push keyboard scope
  usePageFooter({ shortcuts: EXAMPLE_SHORTCUTS });  // 2. Set footer hints
  useBackHandler();                                 // 3. Bind Escape to goBack

  // Screen content...
}
```

### Required Hooks

| Hook | Purpose | Side Effect |
|------|---------|-------------|
| `useScope(name)` | Pushes a keyboard scope on mount, pops on unmount | Ensures this screen's hotkeys are dispatched exclusively |
| `usePageFooter(opts)` | Sets the footer shortcut hints | Clears shortcuts on unmount |
| `useBackHandler()` | Binds Escape to `goBack()` | Enables Escape-to-go-back navigation |

## Screen Inventory

The CLI has 13 screens across 4 groups:

### Top-Level

| Screen | Route | Description |
|--------|-------|-------------|
| HomeScreen | `home` | Main menu with context sidebar, provider status, and action menu |
| OnboardingScreen | `onboarding` | First-run wizard for provider setup |
| ReviewScreen | `review` | Code review flow: progress, results, and issue detail |
| HistoryScreen | `history` | Past review runs with timeline and insights |
| HelpScreen | `help` | Keyboard shortcut reference and help text |

### Settings

| Screen | Route | Description |
|--------|-------|-------------|
| SettingsHubScreen | `settings` | Settings menu with numbered hotkeys |
| ThemeScreen | `settings/theme` | Palette selector using RadioGroup |
| ProvidersScreen | `settings/providers` | AI provider list with API key management |
| StorageScreen | `settings/storage` | Config storage location selector |
| AnalysisScreen | `settings/analysis` | Review analysis options |
| AgentExecutionScreen | `settings/agent-execution` | Agent execution preferences |
| DiagnosticsScreen | `settings/diagnostics` | System diagnostics and debug info |
| TrustPermissionsScreen | `settings/trust-permissions` | Project trust capabilities |

## Screen Composition Example

The SettingsHubScreen demonstrates a typical screen structure:

```tsx
export function SettingsHubScreen(): ReactElement {
  useScope("settings-hub");
  usePageFooter({ shortcuts: SETTINGS_SHORTCUTS });
  useBackHandler();

  const { navigate } = useNavigation();

  const onSelect = (id: string) => {
    const screen = SETTINGS_ROUTE_MAP[id];
    if (screen) {
      navigate({ screen } as Route);
    }
  };

  return (
    <Panel>
      <Panel.Content>
        <Box flexDirection="column" gap={1}>
          <SectionHeader>Settings</SectionHeader>
          <Menu variant="hub" onSelect={onSelect}>
            {SETTINGS_MENU_ITEMS.map((item, index) => (
              <Menu.Item
                key={item.id}
                id={item.id}
                hotkey={index + 1}
                value={item.description}
              >
                {item.label}
              </Menu.Item>
            ))}
          </Menu>
        </Box>
      </Panel.Content>
    </Panel>
  );
}
```

Key patterns visible here:

- Three lifecycle hooks called first
- Navigation config (`SETTINGS_ROUTE_MAP`) kept as a static mapping
- Menu items driven by a config array (`SETTINGS_MENU_ITEMS`)
- `Panel` and `SectionHeader` provide structural framing
- `Menu` with `variant="hub"` provides numbered navigation

## Notes

- Screen components are located in `apps/cli/src/app/screens/`. Settings sub-screens are in `apps/cli/src/app/screens/settings/`.
- Shortcut arrays are defined in `apps/cli/src/config/navigation.ts` alongside menu item definitions.
- The `usePageFooter` hook runs its effect only on mount/unmount. Shortcuts are expected to be static per screen.
- Screens should not import from other screens. Shared logic belongs in feature modules or hooks.
