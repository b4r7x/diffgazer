## Overview

The CLI theme system provides a set of named color tokens (`CliColorTokens`) that all components consume via the `useTheme()` hook. Three palettes ship by default: dark, light, and high-contrast. The active palette is auto-detected from the terminal environment or can be overridden with the `--theme` flag.

## CliColorTokens

Defined in `apps/cli/src/theme/palettes.ts`:

```ts
export interface CliColorTokens {
  // Primitive
  bg: string;
  fg: string;
  blue: string;
  violet: string;
  green: string;
  red: string;
  yellow: string;
  border: string;
  muted: string;
  // Semantic
  success: string;
  warning: string;
  error: string;
  info: string;
  accent: string;
  // Domain (severity)
  severityBlocker: string;
  severityHigh: string;
  severityMedium: string;
  severityLow: string;
  severityNit: string;
  // Domain (status)
  statusRunning: string;
  statusComplete: string;
  statusPending: string;
}
```

Tokens are organized in three tiers:

| Tier | Tokens | Purpose |
|------|--------|---------|
| Primitive | `bg`, `fg`, `blue`, `violet`, `green`, `red`, `yellow`, `border`, `muted` | Raw color values |
| Semantic | `success`, `warning`, `error`, `info`, `accent` | Intent-based colors |
| Domain | `severityBlocker..Nit`, `statusRunning..Pending` | Diffgazer-specific meanings |

## Auto-Detection

When `initialTheme` is `"auto"` (the default), the provider checks the `COLORFGBG` environment variable. This variable is set by many terminal emulators (e.g., iTerm2, rxvt):

- If the last segment of `COLORFGBG` (semicolon-separated) is a number <= 6, the terminal is considered light-background, and the light palette is used.
- Otherwise, the dark palette is used.

If `COLORFGBG` is not set, dark is the fallback.

## useTheme Hook

```ts
import { useTheme } from "../../theme/theme-context.js";

function MyComponent() {
  const { tokens, themeName, setTheme } = useTheme();

  // tokens: CliColorTokens -- current palette values
  // themeName: string -- "dark", "light", or "high-contrast"
  // setTheme: (name: string) => void -- switch palette at runtime
}
```

### Return Values

| Value | Type | Description |
|-------|------|-------------|
| `tokens` | `CliColorTokens` | Current palette's color tokens |
| `themeName` | `string` | Name of the active palette |
| `setTheme` | `(name: string) => void` | Switch to a different palette by name |

## CliThemeProvider

Wrap the component tree in `CliThemeProvider` to make `useTheme()` available:

```tsx
<CliThemeProvider initialTheme="auto">
  {children}
</CliThemeProvider>
```

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `initialTheme` | `string` | -- | Palette name or `"auto"` for auto-detection |
| `children` | `ReactNode` | -- | App content |

## Runtime Theme Switching

The settings theme screen uses `setTheme()` to switch palettes without restarting:

```tsx
const { setTheme } = useTheme();
setTheme("high-contrast"); // Immediate re-render with new palette
```

## Notes

- All color values are hex strings compatible with Ink's `color` and `backgroundColor` props.
- Components should never hardcode color strings. Always use `tokens.{name}`.
- The `palettes` record maps palette names to token objects: `{ dark, light, "high-contrast" }`.
