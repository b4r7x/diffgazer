## High

### F206 — [NEW] [a11y] aria-hidden='false' escape hatch not handled in focusable detection

- **file:line** — `libs/keys/src/dom/focusable.ts:36-37`
- **What** — The isAriaHidden() function uses element.closest('[aria-hidden="true"]') to check if an element or any ancestor has aria-hidden=true, but does not account for the ARIA semantics where aria-hidden="false" on a descendant should re-expose it, overriding the ancestor's aria-hidden=true.
- **Why** — Ignoring the `aria-hidden="false"` override misclassifies re-exposed elements as hidden, so focus/tabbable detection can skip elements that are actually accessible.
- **How** — Modify isAriaHidden() to walk from the element up to the root, stopping at the first element that explicitly sets aria-hidden (either true or false). If the first explicit setting is false, the element is not aria-hidden; if true, it is. Alternatively, check the computed aria-hidden value on the element itself using getComputedAccessibleDescription or a direct attribute walk.
- **Effort** — low

### F321 — [NEW] [a11y] Anchor elements with aria-disabled lack keyboard navigation blocking

- **file:line** — `libs/ui/registry/ui/button/button.tsx:180-197`
- **What** — Anchor elements rendered with aria-disabled prop for disabled state, but no role="button" is set. When aria-disabled="true", screen readers may still present the link as operable, and keyboard users can still Tab to and interact with the link using Enter.
- **Why** — A disabled anchor that remains Tab-reachable and activatable contradicts its disabled state, letting keyboard and screen-reader users trigger a control meant to be inert.
- **How** — Either add role="button" to the anchor element with aria-disabled, or render disabled anchor elements as <span> or <button disabled> instead. If keeping <a>, ensure tabIndex is always -1 when aria-disabled is true, and test with keyboard navigation to confirm it's skipped by Tab key.
- **Effort** — medium

## Medium

### F81 — [NEW] [architecture] Checkbox/Radio components use hidden native inputs with aria-hidden for state synchronization

- **file:line** — `libs/ui/registry/ui/checkbox/checkbox.tsx:173-191`
- **What** — Checkbox and Radio components render a hidden native <input> with aria-hidden="true" alongside the custom div-based control, primarily for form reset and native validation semantics
- **Why** — A redundant aria-label and a refocus-on-invalid handler on the hidden input can duplicate or compete with the visible control's accessible naming and focus management.
- **How** — Review whether the hidden input's aria-label (line 185 checkbox.tsx, 217 radio.tsx) is necessary when the visible control already has aria-labelledby/aria-label. Consider whether the native validation handler (onInvalid) could directly set aria-invalid on the visible control without refocusing
- **Effort** — medium

### F86 — [NEW] [a11y] Spinner component in buttons uses aria-hidden but provides no visible loading state alternative for decorated buttons

- **file:line** — `libs/ui/registry/ui/button/button.tsx:100-129`
- **What** — When a Button is loading, the spinner is rendered with aria-hidden="true" and button content is wrapped in sr-only. If a consumer uses bracket=true or custom children, the loading state may not be visually distinct to users
- **Why** — Sighted users relying on visual cues may not perceive the loading state when the spinner is the only indicator and content is visually hidden, leaving the busy state ambiguous.
- **How** — Document that buttons with loading=true should have CSS rules (e.g., hover:disabled, opacity change) to visually convey the loading state, separate from aria-busy. Consider adding a data-loading attribute to the button element for easier CSS targeting in consuming apps
- **Effort** — low

### F322 — [NEW] [a11y] Checkbox and Radio hidden native inputs do not inherit aria-invalid state

- **file:line** — `libs/ui/registry/ui/checkbox/checkbox.tsx:173-191`
- **What** — Checkbox (and similar Radio) renders a hidden native <input type="checkbox" aria-hidden=true> only when name or required is set. This input does not receive aria-invalid or aria-describedby even when the custom checkbox has those properties. The aria-label on the hidden input does not match the container's aria-labelledby strategy.
- **Why** — Invalid and description state present on the visible control is not mirrored to the form-participating input, so the accessible validity/description contract is inconsistent between the two.
- **How** — Pass aria-invalid and aria-describedby to the hidden native input element, matching the custom checkbox's aria attributes. Alternatively, document that the hidden input is for validation only and the accessible state is on the container.
- **Effort** — low

### F323 — [NEW] [a11y] Button anchor variant does not remove href on disabled state

- **file:line** — `libs/ui/registry/ui/button/button.tsx:169-198`
- **What** — When Button rendered as <a> with as="a" and isDisabled=true, the href prop is still passed to the anchor. This allows screen reader users to potentially activate the link via Enter (depending on browser and assistive tech), even though onClick preventDefault is in place. The href should be removed or set to undefined when disabled.
- **Why** — A disabled anchor that retains its href stays navigable through assistive tech and Enter, so the disabled state is not reliably enforced across browsers.
- **How** — Do not pass href when isDisabled is true, or use href={isDisabled ? undefined : originalHref}. Alternatively, render disabled anchor buttons as <button> elements instead.
- **Effort** — low

### F325 — [NEW] [a11y] RadioGroup required validation uses hidden checkbox instead of native radio

- **file:line** — `libs/ui/registry/ui/radio/radio-group.tsx:260-276`
- **What** — RadioGroup renders a hidden <input type="checkbox" required> to validate that a radio is selected, but it should use <input type="radio" required> to properly integrate with browser and OS native radio validation semantics. The hidden checkbox does not semantically represent the radiogroup.
- **Why** — Using a checkbox to back radiogroup validation mismatches native radio semantics, so browser/OS required-radio validation does not represent the group correctly.
- **How** — Change the hidden input type from 'checkbox' to 'radio' to match the radiogroup semantics, or refactor validation to not rely on a native input at all (validate via onInvalid callback on RadioGroup itself).
- **Effort** — medium

## Low

### F83 — [NEW] [a11y] Avatar component provides empty alt text for all images (decorative contract not explicit)

- **file:line** — `libs/ui/registry/ui/avatar/avatar-image.tsx:38`
- **What** — AvatarImage defaults alt to an empty string (`alt={alt ?? ""}`) without documenting that avatars are decorative images and text labels must be provided at the avatar context level (e.g., in the containing card or profile section)
- **Why** — Without an explicit decorative contract, consumers may not provide a text label elsewhere, leaving the avatar's identity unannounced to screen readers.
- **How** — Add a JSDoc comment to AvatarImage explaining: 'Avatar images are treated as decorative; the user is named elsewhere in the containing component. Do not override alt unless the Avatar itself must announce the user name.' Update avatar tests to verify an avatar-containing card has an accessible name
- **Effort** — low

### F84 — [NEW] [a11y] Dialog without title or aria-label logs a warning but does not enforce accessible naming

- **file:line** — `libs/ui/registry/ui/dialog/dialog-content.tsx:136-141`
- **What** — If a Dialog is opened without a <Dialog.Title>, aria-label, or aria-labelledby, resolveAccessibleName() falls back to a generic 'Dialog' label and logs a console.warn in development
- **Why** — A soft warning with a generic-name fallback lets unnamed dialogs ship, so the accessible-name requirement is not actually enforced.
- **How** — Escalate the warning to a thrown error (or a data-testid marker) in development/test environments, making it a hard contract. Add a JSDoc example showing the three naming paths (Title, aria-label, aria-labelledby) and a test that verifies all three paths work
- **Effort** — low

### F85 — [NEW] [a11y] PopoverContent role assignment depends on triggerMode without documenting the contract

- **file:line** — `libs/ui/registry/ui/popover/popover-content.tsx:74-75`
- **What** — PopoverContent automatically assigns role="tooltip" for hover-triggered popovers and role="dialog" (or custom role) for click-triggered popovers, but this logic is implicit and not documented
- **Why** — Implicit role-by-triggerMode behavior is easy to misuse, since consumers cannot tell which ARIA role they get or how to override it.
- **How** — Add a JSDoc comment to PopoverContent explaining the role assignment logic: 'Role is determined by triggerMode: hover → tooltip (read-only hint), click → dialog or custom role (interactive). Override with the role prop if needed.' Add an example showing how to override role for a custom hover-triggered menu
- **Effort** — low

### F324 — [NEW] [a11y] SearchInput focus ring styled in CSS, may not respect prefers-reduced-motion

- **file:line** — `libs/ui/registry/ui/search-input/search-input.tsx:10-25`
- **What** — The searchInputVariants CVA includes focus-within:border-foreground transition, but no CSS rule respects prefers-reduced-motion media query. Focus indicators are styled with transitions, which may annoy users who have enabled reduced-motion preference.
- **Why** — Unconditional focus transitions ignore the user's reduced-motion preference, applying motion to users who have asked to avoid it.
- **How** — Wrap transition classes in @media (prefers-reduced-motion: no-preference) or add a separate CSS rule that overrides transitions to none when prefers-reduced-motion: reduce is active.
- **Effort** — low

### F326 — [NEW] [a11y] CheckboxGroup required validation input has duplicate aria-labelledby

- **file:line** — `libs/ui/registry/ui/checkbox/checkbox-group.tsx:209-227`
- **What** — CheckboxGroup renders a hidden required input with both aria-label AND aria-labelledby. If ariaLabelledBy is set, aria-label will be ignored; if ariaLabelledBy is not set, aria-label is used. This mixing of labeling strategies is redundant and may confuse accessibility tools.
- **Why** — Supplying both labeling strategies at once is redundant and can confuse accessibility tooling about which name applies.
- **How** — Use aria-labelledby only if ariaLabelledBy is defined, otherwise use aria-label. Example: aria-labelledby={ariaLabelledBy || undefined} and remove aria-label, or keep aria-label only if ariaLabelledBy is not set.
- **Effort** — low

### F327 — [NEW] [a11y] InputGroup prefix and suffix aria-hidden logic does not handle conditional content

- **file:line** — `libs/ui/registry/ui/input/input-group.tsx:20-62`
- **What** — InputGroup marks prefix/suffix as aria-hidden="true" only if they are plain strings or numbers (isPlainDecorativeAffix). However, a ReactNode can be a string, number, or a component that is actually decorative (e.g., an icon). If a component is passed, it will not be marked aria-hidden even if it is visually decorative, potentially exposing it to screen readers.
- **Why** — Decorative component affixes fall outside the plain-string check, so they remain exposed to screen readers and add noise to the input's accessible output.
- **How** — Either (a) require decorative components to wrap themselves in aria-hidden, (b) add an optional ariaHidden prop to InputGroup to let consumers declare decorative status, or (c) always mark prefix/suffix aria-hidden and require interactive content to manage its own accessibility.
- **Effort** — low
