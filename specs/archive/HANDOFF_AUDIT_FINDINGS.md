# Handoff Audit Findings

Read-only SOTA audit summary for public handoff readiness.

Current verdict: not ready for broad public handoff yet. The project is close in package structure and architecture, but there are blockers in local app security, shadcn/copy registry correctness, and release/deploy supply-chain safety.

## Findings

| Severity | Area | Evidence | Issue | Why this is a real problem | Task |
|---|---|---|---|---|---|
| Critical | CLI security | `cli/diffgazer/src/lib/shutdown-token.ts:5`, `cli/diffgazer/src/lib/servers/embedded-server.ts:90`, `cli/server/src/app.ts:75` | JS-readable bearer token grants full local API control. | Any XSS, browser extension, or frontend bug can read repositories, mutate config/trust, use stored API keys, and shut down the CLI. | Move API auth to an `HttpOnly`, `SameSite=Strict` session cookie and split shutdown authorization from API authorization. |
| Critical | CLI browser hardening | `cli/server/src/app.ts:61`, `cli/diffgazer/src/lib/servers/embedded-server.ts:97` | Strict CSP and full secure-header model are missing. | Without CSP, the JS-readable token issue has much higher impact because injected script can freely read the token. | Add Hono `secureHeaders()` plus explicit CSP. Remove inline secret bootstrapping or replace it with a nonce-based flow that does not expose secrets. |
| Critical | Direct shadcn/copy | `libs/ui/public/r/use-navigation.json:12`, `libs/ui/scripts/transform-public-registry-keys-imports.ts:40` | Four public UI hook shims re-export `@diffgazer/keys` in copy/direct-shadcn output. | Copy/direct-shadcn install should copy dependencies through registry dependencies. Current output silently requires an npm package that is not declared as a dependency. | Rewrite `export ... from "@diffgazer/keys"` in public registry output, or consistently declare the npm dependency. |
| Critical | Release supply chain | `.github/workflows/release.yml:3`, `.github/workflows/release.yml:30` | Release workflow is triggered by successful readiness but does not checkout that readiness run SHA. | A later `main` commit can be published even though a different commit passed readiness/E2E. | Checkout `github.event.workflow_run.head_sha` in the release job. |
| Critical | Deploy safety | `.github/workflows/deploy.yml:3`, `.github/workflows/deploy.yml:76` | Deploy can run independently of readiness and masks webhook failure with `|| echo`. | CI can appear green while production did not deploy, or while unverified code is deployed. | Gate deploy on readiness success, remove failure masking, and add post-deploy `curl -fI` checks. |
| High | npm publishing | `.github/workflows/release.yml:24`, `.github/workflows/release.yml:54` | Publishing still uses `NPM_TOKEN` even though the workflow has `id-token: write`. | Long-lived npm tokens are weaker supply-chain posture than Trusted Publishing/OIDC and are avoidable for public release. | Configure npm Trusted Publisher, publish via OIDC, and disable package-manager cache in the publish job. |
| High | Secrets at rest | `apps/web/src/features/onboarding/hooks/use-onboarding.ts:22`, `cli/server/src/shared/lib/config/state.ts:124` | File-backed plaintext API key storage is the default onboarding path. | Mode `0600` is good, but API keys are still plaintext on disk. | High | Handoff state | `specs/001-p0-handoff/tasks.md:20`, `specs/001-p0-handoff/EXECUTE.md:260` | P0 handoff spec still lists unchecked `MUST FIX` tasks. | It is unclear whether the spec is stale or the project still has unhandled blockers. This makes handoff ambiguous for the next agent. | Reconcile the P0 checklist with current code and mark each item done, obsolete, or still blocking. |
| High | Docs install URL | `libs/keys/docs/content/getting-started/installation.mdx:41`, `apps/docs/src/lib/consumption-metadata.ts:71` | Keys direct shadcn URL omits the `/r/` prefix. | Public install snippet can fail against the real registry even if local smoke tests accept both path shapes. | Standardize on `https://r.b4r7.dev/r/keys/...`. |
| High | UI accessibility SSR | `libs/ui/registry/ui/field/field.tsx:97`, `libs/ui/registry/ui/field/field.tsx:209` | `Field` ARIA relationships are established only after hydration effects. | Initial SSR/no-JS DOM misses label, description, and error relationships. | High | Select accessibility | `libs/ui/registry/ui/select/select-trigger.tsx:63`, `libs/ui/registry/ui/select/select-search.tsx:55` | Searchable `Select` does not carry the `Field` label to the actual combobox. | Screen readers can see a generic "Search options" combobox instead of the field label and description. | Forward `aria-labelledby` and `aria-describedby` from `Field`/trigger props to the search combobox. |
| High | Docs architecture | `apps/docs/config/docs-libraries.json:28`, `libs/registry/src/docs/sync-operations.ts:289` | Future app docs are not a config-only addition. | `prepare:generated` can omit or erase manual docs if a library/app does not have the artifact-backed contract. | Decide one contract: every docs section is artifact-backed, or docs supports preserved manual namespaces. |
| High | Docs deployment | `apps/docs/vite.config.ts:31`, `apps/docs/src/server.ts:1`, `Dockerfile:50` | Docs are a TanStack Start/Nitro server deployment, not a plain static site. | Deploying as static hosting can break server functions, search, and runtime data paths. | Deploy `docs.b4r7.dev` as a Node/TanStack Start app, or remove/convert runtime server functions before static hosting. |
| High | Registry validation | `libs/registry/src/shadcn/validate.ts:105`, `libs/ui/public/r/diff-view.json:7` | Hidden public registry items are not freshness-validated. | Visible registry items depend on hidden JSON files. | Medium | CI smoke coverage | `scripts/monorepo/smoke-shadcn-install.mjs:28`, `scripts/monorepo/smoke-shadcn-install.mjs:477` | Shadcn smoke does not direct-install every public JSON. | The current broken hidden hook shim case is missed by smoke coverage. | Add direct-install smoke for all public item JSONs or at least all hidden transitives. |

| Medium | API abuse | `cli/server/src/features/review/router.ts:30`, `cli/server/src/shared/lib/ai/client.ts:141` | Expensive token-authenticated routes have no rate limit or in-flight cap. | A token leak or UI loop can spam AI providers using stored credentials. | Add per-process rate limits and concurrency caps for review creation, drilldown, model fetching, and context refresh. |
| Medium | Review input validation | `cli/server/src/features/review/schemas.ts:27` | `files[]` accepts loose strings without count/length/path policy. | Weird git pathspecs and large lists can create correctness or DoS risk. | Add max item count, max length, no NUL, and explicit relative path/pathspec policy. |
| Medium | CI dirty check | `.github/workflows/release-readiness.yml:48`, `package.json:22` | Dirty check runs before `verify`, while `verify` runs artifact preparation. | Tracked public registry drift created during verify may not be caught by CI. | Repeat `git status --short` after `verify` or move the dirty check later. |
| Medium | Coverage gates | `turbo.json:25`, `libs/ui/vitest.config.ts:14` | Coverage thresholds exist but release scripts run plain `vitest run`. | The repo appears to have coverage gates, but they are not enforced. | Add a coverage script to readiness/release or remove dead coverage threshold expectations. |
| Medium | Lint gates | `package.json:21`, `package.json:23` | Root `check` is not part of release readiness. | Static/lint regressions can ship even when release gates are green. | Wire real lint/check into readiness and release gates. |

## Required Verification Before Public Handoff

Run these after fixes:

```bash
pnpm run prepare:artifacts
pnpm run validate:artifacts:check
DIFFGAZER_SKIP_ARTIFACT_PREPARE=1 pnpm exec turbo run type-check
DIFFGAZER_SKIP_ARTIFACT_PREPARE=1 pnpm exec turbo run test
DIFFGAZER_SKIP_ARTIFACT_PREPARE=1 pnpm exec turbo run test:types
DIFFGAZER_SMOKE_STRICT_SKIPS=1 pnpm run smoke
pnpm run verify:monorepo
pnpm run release-check
git diff --check
```
